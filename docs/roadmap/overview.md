# Muブラウザ 開発ロードマップ

## プロジェクト現状

### 実装状況

Muプロジェクトは現在、**設計フェーズから実装フェーズへの移行準備が整った状態**です。

- **コードベース**: 初期テンプレート段階
  - フロントエンド: React 19 + TypeScript + Tailwind CSS 4の基本構成
  - バックエンド: Tauri 2.x + Rustの基本構成
  - 実装済み機能: テンプレートコマンド（`greet`）のみ
- **ドキュメント**: 充実した設計・仕様書が完備
  - 要件定義（requirements.md）
  - コーディングガイドライン群（docs/rules/配下）
  - プロジェクト指針（CLAUDE.md）

### README.mdとの関係

README.mdのロードマップセクションでは多くの機能が「実装済み」とマークされていますが、これらは**実装予定の機能**を示しています。実際の実装はこれから段階的に行います。

このドキュメント（overview.md）と各フェーズファイルが、実装の具体的な計画とタスク管理を担います。

## フェーズ概要

Muブラウザの開発は4つのフェーズに分けて段階的に進めます。各フェーズは明確な目標と成果物を持ち、前のフェーズの成果を基盤として構築されます。

### Phase 1: 基盤構築

**目標**: ブラウザとして最低限機能する状態を実現

WebViewの統合、基本的なUI、ナビゲーション機能を実装します。このフェーズでユーザーはURLを入力し、Webページを閲覧し、基本的なナビゲーション（戻る/進む/リロード）を行えるようになります。

**主要機能**:
- Tauri WebviewWindowの統合
- 基本的なウィンドウレイアウト
- 暫定的なアドレスバー（Phase 2でコマンドパレットに置き換え）
- ナビゲーション機能（戻る/進む/リロード/ホーム）

**成果物**: v0.1.0 - 基本的なWebページ表示とナビゲーション

**推定期間**: 4-6週間

[詳細はこちら →](./phase-1-foundation.md)

---

### Phase 2: コア機能

**目標**: Muの差別化要素となるコマンドパレットとタブ管理を実装

キーボード中心の操作体系を確立し、複数タブの管理機能を実装します。このフェーズでMuの独自性が明確になり、従来のブラウザとの差別化が実現されます。

**主要機能**:
- コマンドパレット（Ctrl+K / Cmd+K）
- URL入力とコマンド実行の統合インターフェース
- 垂直タブ管理（作成/削除/切り替え）
- キーボードショートカット体系

**成果物**: v0.2.0 - コマンドパレットで操作可能、複数タブ管理

**推定期間**: 3-4週間

[詳細はこちら →](./phase-2-core-features.md)

---

### Phase 3: 高度な機能

**目標**: カスタマイズ性と生産性向上機能を追加

サイドバーシステム、広告ブロック、設定システムを実装します。ユーザーはブラウザを自分の好みにカスタマイズし、より快適な閲覧体験を得られます。

**主要機能**:
- サイドバーシステム（左右配置、固定/自動隠蔽モード）
- 広告ブロック（ドメインベースのブロックリスト）
- 設定システム（永続化、設定UI）
- テーマ切り替え（ライト/ダーク）

**成果物**: v0.3.0 - フル機能実装、カスタマイズ可能

**推定期間**: 3-4週間

[詳細はこちら →](./phase-3-advanced-features.md)

---

### Phase 4: 磨き上げ

**目標**: パフォーマンス目標達成とリリース準備

パフォーマンスの測定と最適化、UX改善、ドキュメント整備、テストを行います。このフェーズで一般リリースに向けた品質を確保します。

**主要機能**:
- パフォーマンス最適化（起動時間、メモリ使用量、バンドルサイズ）
- UX改善（エッジケース対応、エラーハンドリング）
- ドキュメント整備（ユーザーガイド、開発者向けドキュメント）
- E2Eテストとユーザビリティテスト

**成果物**: v1.0.0 - 一般リリース準備完了

**推定期間**: 2-3週間

[詳細はこちら →](./phase-4-polish.md)

## 依存関係の可視化

各フェーズは前のフェーズの成果物を基盤として構築されます。

```
Phase 1: 基盤構築（WebView、ナビゲーション）
    ↓
Phase 2: コア機能（コマンドパレット、タブ管理）
    ↓
Phase 3: 高度な機能（サイドバー、広告ブロック、設定）
    ↓
Phase 4: 磨き上げ（パフォーマンス最適化、品質向上）
```

**並行開発の可能性**:
- Phase 2内: コマンドパレットとタブ管理は独立して開発可能
- Phase 3内: サイドバーと広告ブロックは独立して開発可能

## 開発原則

Muブラウザの開発において、以下の原則を常に念頭に置きます。

### 1. ミニマリズム優先

機能を追加する前に「本当に必要か？」を自問します。80/20ルールに従い、20%の機能で80%のユースケースをカバーすることを目指します。

- UIは3-4色に制限
- アニメーションは最小限
- 設定項目は必要最小限

### 2. パフォーマンス目標

以下の目標を常に意識し、実装します。

- **起動時間**: 1秒以内
- **バンドルサイズ**: 200KB以内（minified + gzipped）
- **メモリ使用量**: 100MB以内（アイドル時）

### 3. キーボードファースト

全ての主要機能はキーボードのみで完結できるようにします。マウスは補助的な入力手段として位置づけます。

- グローバルショートカットは10個以内に制限
- コマンドパレットで全機能にアクセス可能

### 4. 依存ライブラリの慎重な選択

100行以内で実装できる機能は自前実装を優先します。新しいパッケージの追加は慎重に検討します。

## 実装時の注意事項

### ディレクトリ構造の遵守

実装時は必ず`docs/rules/file-structure.md`で定義されたディレクトリ構造に従ってください。

**フロントエンド**:
```
src/
├── components/        # 共通UIコンポーネント
├── features/          # 機能別モジュール
├── hooks/             # 共通hooks
├── types/             # 型定義
└── assets/            # 共通assets
```

**バックエンド**:
```
src-tauri/src/
├── modules/           # 機能別モジュール
│   └── [機能名]/
│       ├── mod.rs
│       ├── commands.rs
│       ├── models.rs
│       └── utils.rs
└── lib.rs
```

### コーディングガイドラインの参照

実装時は以下のガイドラインを参照してください。

- `docs/rules/coding-guidelines.md` - TypeScript/React、Rust/Tauriの規約
- `docs/rules/minimalism-implementation.md` - 機能優先順位の判断基準
- `docs/rules/ui-design-guidelines.md` - UI実装パターン
- `docs/rules/performance-guidelines.md` - パフォーマンス測定方法
- `docs/rules/error-handling.md` - エラー処理の方針

### 実装ノートの活用

実装中に得た知見、トラブルシューティング、解決策は`implementation-notes.md`に記録してください。これにより、チームやコミュニティでの知識共有が促進されます。

[実装ノートはこちら →](./implementation-notes.md)

## フェーズ別の実装順序

### Phase 1の進め方

1. バックエンド: WebView統合 → ナビゲーション機能
2. フロントエンド: 基本レイアウト → アドレスバー → ナビゲーションUI
3. 統合テスト: URL表示、ナビゲーション動作確認

### Phase 2の進め方

1. キーボードショートカット基盤の実装
2. コマンドパレット（Phase 1のアドレスバーを置き換え）
3. タブ管理システム（バックエンド → フロントエンド）
4. 統合テスト: キーボード操作、タブ切り替え

### Phase 3の進め方

1. 設定システム（他機能の基盤として最初に実装）
2. サイドバーシステム
3. 広告ブロック機能
4. 統合テスト: カスタマイズ、フィルタリング

### Phase 4の進め方

1. パフォーマンス測定 → ボトルネック特定 → 最適化
2. バグ修正とエッジケース対応
3. ドキュメント作成
4. E2Eテストとユーザビリティテスト

## 既存ドキュメントとの関係性

### requirements.md との違い

- **requirements.md**: 「何を作るか」（What） - 要件定義、機能仕様
- **docs/roadmap/**: 「いつ、どう作るか」（When, How） - 実装計画、タスク管理

ロードマップでは要件の詳細を繰り返さず、実装の順序と手順に集中します。詳細な機能仕様はrequirements.mdを参照してください。

### README.md との関係

README.mdのロードマップセクションは簡潔な概要を提供します。詳細な実装計画とタスク管理はこのdocs/roadmap/配下のドキュメントで行います。

README.mdは外部向けの概要、docs/roadmap/は開発者向けの詳細という位置づけです。
